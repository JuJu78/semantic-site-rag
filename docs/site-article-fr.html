Un site web n'est pas seulement une collection de pages. C'est un corpus.

Et si ce corpus est bien structuré, il peut devenir une base de connaissance interrogeable par ChatGPT, Codex ou n'importe quel assistant connecté à Supabase.

Dans ce tutoriel, nous allons construire un RAG SEO vivant pour un site web : une base Supabase qui contient les URLs du site, les contenus, les chunks, les embeddings, les liens internes, les ancres de lien et quelques signaux utiles pour piloter le maillage interne.

L'objectif n'est pas de faire une démo RAG de plus. L'objectif est de construire un système utile pour répondre à des questions concrètes :

  • quelles pages ont trop peu de liens internes ?
  • quelles pages sont proches sémantiquement et devraient être reliées ?
  • quelles ancres pointent vers une URL donnée ?
  • quelles pages sont éloignées du thème principal du site ?
  • quelles URLs ont disparu du sitemap ?
  • quelles pages ont changé depuis la dernière mise à jour ?
  • quelles pages faut-il réembedder, et lesquelles faut-il laisser tranquilles pour éviter les coûts inutiles ?

Le repo utilisé dans ce guide est disponible ici :

https://github.com/JuJu78/semantic-site-rag

Ce que nous allons construire

Nous allons créer une base de données Supabase capable de stocker :

  • une table de pages, avec une URL par ligne ;
  • une table de chunks, avec plusieurs passages par page ;
  • un embedding pour chaque page ;
  • un embedding pour chaque chunk ;
  • les liens internes détectés dans chaque page ;
  • les ancres utilisées dans ces liens ;
  • un PageRank interne ;
  • un hash de contenu pour ne réembedder que les pages modifiées ;
  • des informations de présence ou d'absence dans le sitemap.

Une fois cette base en place, vous pouvez connecter Supabase à ChatGPT ou Codex et demander à l'assistant de raisonner sur l'état réel de votre site.

Pourquoi Supabase est intéressant pour ce cas d'usage

Supabase combine PostgreSQL, une API simple, et l'extension pgvector.

En clair, cela permet de stocker des données classiques, comme des URLs ou des ancres, mais aussi des vecteurs sémantiques. Ces vecteurs sont les embeddings générés à partir de vos contenus.

C'est précisément ce qu'il nous faut pour un RAG SEO :

  • PostgreSQL pour les pages, les chunks et les liens ;
  • pgvector pour la recherche sémantique ;
  • le connecteur Supabase de ChatGPT ou Codex pour interroger la base ;
  • un script Python pour garder les données à jour.

Pour un petit ou moyen site, le plan gratuit de Supabase permet déjà d'aller loin. Il faut évidemment surveiller les limites, mais pour apprendre et prototyper, c'est très confortable.

Pré-requis

Vous aurez besoin de :

  • un compte Supabase ;
  • une clé API OpenAI ;
  • Python installé sur votre machine ;
  • un terminal ;
  • un sitemap XML de votre site.

Si vous débutez complètement avec Supabase, ce n'est pas un problème. Nous allons dérouler les étapes dans l'ordre.

Étape 1 : créer un projet Supabase

Allez sur Supabase :

https://supabase.com/

Créez un compte, puis créez un nouveau projet.

Supabase vous demandera un nom de projet, un mot de passe de base de données et une région d'hébergement.

Une fois le projet créé, ouvrez :

Project Settings > API

Récupérez deux informations :

  • Project URL
  • service_role key

La clé service_role donne beaucoup de droits. Ne la mettez jamais dans un repo public. Elle doit rester dans votre fichier .env.

Étape 2 : activer pgvector

Dans Supabase, ouvrez le SQL Editor et exécutez le fichier :

sql/001_enable_pgvector.sql

Il contient :

create extension if not exists vector;

Cette extension permet à PostgreSQL de stocker et comparer des embeddings.

Étape 3 : créer les tables

Exécutez ensuite :

sql/002_create_tables.sql

Ce script crée les tables principales.

rag_pages contient une ligne par URL. On y stocke le titre, la description, le contenu complet, le hash du contenu, l'embedding de page, le PageRank et les informations de sitemap.

rag_chunks contient plusieurs lignes par URL. Chaque ligne correspond à un morceau de contenu, avec son propre embedding.

rag_links contient les liens internes détectés, avec l'URL source, l'URL cible et l'ancre.

rag_refresh_runs sert à suivre les mises à jour.

Exécutez enfin :

sql/003_create_match_functions.sql

Ce fichier ajoute les fonctions SQL de recherche vectorielle, comme match_rag_pages et match_rag_chunks.

Étape 4 : installer le repo

Dans votre terminal :

git clone https://github.com/JuJu78/semantic-site-rag.git
cd semantic-site-rag

Créez un environnement virtuel :

python -m venv .venv

Sur Windows :

.venv\Scripts\activate

Sur macOS ou Linux :

source .venv/bin/activate

Installez les dépendances :

pip install -r requirements.txt

Étape 5 : configurer le fichier .env

Copiez le fichier d'exemple :

copy .env.example .env

Sur macOS ou Linux :

cp .env.example .env

Remplissez ensuite les variables :

OPENAI_API_KEY=sk-...
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
SITE_BASE_URL=https://example.com
CHUNK_SIZE=1200
CHUNK_OVERLAP=180
REQUEST_TIMEOUT=30
CRAWL_DELAY_SECONDS=1.0
USER_AGENT=SemanticSiteRAG/0.1 (+https://example.com)

Remplacez https://example.com par votre domaine.

Étape 6 : lancer le premier crawl

Pour crawler un sitemap :

python scripts/ingest_site.py --sitemap https://example.com/sitemap.xml

Pour plusieurs sitemaps :

python scripts/ingest_site.py --sitemap https://example.com/sitemap-fr.xml --sitemap https://example.com/sitemap-en.xml

Pour tester sur une seule URL du sitemap :

python scripts/ingest_site.py --sitemap https://example.com/sitemap.xml --limit 1

Le script va lire le sitemap, récupérer les URLs, télécharger les pages, extraire le contenu principal, générer les embeddings, découper les pages en chunks et stocker les liens internes trouvés.

Étape 7 : injecter une URL spécifique

Cas typique : vous venez de publier un nouvel article et il n'est pas encore dans Supabase.

Vous pouvez l'injecter directement :

python scripts/ingest_site.py --url https://example.com/nouvelle-page

Ce mode est utile après publication. La page existe en ligne, donc le script peut la crawler, extraire son contenu, stocker ses chunks, générer ses embeddings et enregistrer les liens internes qu'elle contient.

Étape 8 : trouver des opportunités de maillage depuis un brouillon

Autre cas très utile : vous avez un contenu brut, mais il n'est pas encore publié.

Dans ce cas, il ne faut pas forcément le stocker dans Supabase. On veut seulement savoir quelles pages existantes sont les plus proches sémantiquement.

Le repo contient donc un script séparé :

python scripts/find_link_opportunities.py --text-file draft.md

Ce script lit le brouillon, calcule un embedding temporaire, interroge les pages et chunks déjà stockés dans Supabase, puis génère :

reports/link_opportunities.json

Point important : le texte brut et son embedding ne sont pas insérés dans Supabase.

Vous obtenez un rapport avec les pages les plus proches, les chunks qui justifient la proximité et les URLs candidates pour le maillage interne.

Vous pouvez ensuite demander à ChatGPT ou Codex :

À partir de reports/link_opportunities.json, propose-moi les liens internes à ajouter dans mon brouillon.
Pour chaque recommandation, donne l'URL cible, l'ancre proposée et la justification.

Étape 9 : calculer le PageRank interne

Une fois les liens internes stockés, vous pouvez calculer un PageRank interne :

python scripts/compute_pagerank.py

Ce score n'est pas le PageRank Google. C'est un indicateur interne qui permet de comparer les pages selon le poids transmis par votre propre maillage.

Étape 10 : auditer le maillage interne

Lancez :

python scripts/audit_site.py --min-incoming 3

Le script produit :

reports/site_audit.json

Vous y trouverez notamment les pages avec trop peu de liens internes entrants, les pages les plus éloignées du centroïde sémantique du site, leur PageRank interne et leurs liens entrants/sortants.

Le centroïde sémantique correspond à la moyenne des embeddings de vos pages. Une page très éloignée du centroïde peut être hors sujet, mal extraite, isolée dans un silo ou simplement insuffisamment reliée au reste du corpus.

Ce n'est pas une preuve automatique. C'est un signal d'investigation.

Étape 11 : mettre à jour la base sans tout réembedder

Le piège classique d'un RAG de site web est de tout réembedder tous les jours.

Sur un petit site, ce n'est pas dramatique. Sur un gros site, cela devient vite inutilement coûteux.

Le repo évite cela avec un hash de contenu.

À chaque refresh, le script recrawle la page, recalcule le hash, puis le compare au hash stocké dans Supabase.

Si le contenu n'a pas changé, la page n'est pas réembeddée.

Si le contenu a changé, la page est mise à jour, les anciens chunks sont remplacés et les embeddings sont régénérés.

Lancez :

python scripts/refresh_daily.py --sitemap https://example.com/sitemap.xml

Le script marque aussi les URLs présentes en base mais absentes du sitemap. C'est utile pour repérer les pages supprimées, redirigées ou qui ne devraient plus rester dans votre base RAG.

Étape 12 : automatiser le refresh quotidien

Sur Windows, vous pouvez utiliser le Planificateur de tâches avec une commande de ce type :

D:\path\to\semantic-site-rag\.venv\Scripts\python.exe D:\path\to\semantic-site-rag\scripts\refresh_daily.py --sitemap https://example.com/sitemap.xml

Sur Linux, vous pouvez utiliser cron :

0 6 * * * cd /path/to/semantic-site-rag && /path/to/semantic-site-rag/.venv/bin/python scripts/refresh_daily.py --sitemap https://example.com/sitemap.xml

Le repo contient aussi un exemple GitHub Actions :

.github/workflows/daily-refresh.yml

Dans GitHub, ajoutez les secrets nécessaires :

  • SITEMAP_URL
  • OPENAI_API_KEY
  • SUPABASE_URL
  • SUPABASE_SERVICE_ROLE_KEY

Étape 13 : connecter Supabase à ChatGPT ou Codex

Une fois les données en base, connectez Supabase à ChatGPT ou Codex via le connecteur natif.

L'assistant pourra alors interroger :

  • rag_pages
  • rag_chunks
  • rag_links
  • rag_refresh_runs

Vous pouvez lui demander :

Liste les 20 pages qui ont le moins de liens internes entrants.
Pour chaque page, propose 5 pages sources qui pourraient faire un lien vers elle.
Utilise la proximité sémantique des chunks et les ancres existantes comme indices.

Ou :

Trouve les pages les plus éloignées du centroïde sémantique du site.
Explique si elles semblent hors sujet, isolées ou simplement mal reliées.

Ou encore :

Pour cette URL, liste toutes les ancres internes utilisées.
Dis-moi si les ancres sont trop répétitives, trop vagues ou bien diversifiées.

Pourquoi stocker les pages et les chunks ?

L'embedding de page donne une vision globale d'une URL. Il sert à comparer des pages entre elles, calculer un centroïde et repérer des pages éloignées.

L'embedding de chunk donne une vision plus précise. Il permet de retrouver un passage exact, de proposer un lien interne depuis une section spécifique ou de répondre à une question très ciblée.

Un bon RAG SEO a besoin des deux niveaux.

Ce que ce système permet concrètement

Une fois en place, vous pouvez utiliser cette base pour :

  • trouver les pages sous-maillées ;
  • proposer des liens internes depuis des contenus proches ;
  • auditer les ancres de lien ;
  • repérer les contenus isolés ;
  • identifier des contenus proches qui méritent d'être consolidés ;
  • nettoyer les anciennes URLs absentes du sitemap ;
  • préparer le maillage d'un brouillon avant publication ;
  • garder une base RAG à jour sans exploser les coûts d'embedding.

Il faut toutefois rester rigoureux : une forte proximité sémantique ne prouve pas à elle seule une cannibalisation SEO. Pour parler de cannibalisation, il faut idéalement croiser avec Google Search Console et les requêtes réellement positionnées.

Limites

Ce projet est une base pédagogique et open-source. Il ne remplace pas un crawl SEO complet, une analyse de logs, Google Search Console ou un audit éditorial humain.

Il donne en revanche une fondation très solide pour construire un assistant SEO connecté à l'état réel de votre site.

Autre limite : l'extraction de contenu n'est jamais parfaite. Certains sites très JavaScript, très bloqués ou mal structurés peuvent nécessiter des adaptations.

Conclusion

Le vrai intérêt de ce système n'est pas seulement technique.

Le vrai intérêt, c'est de transformer votre site en corpus exploitable.

Au lieu de demander à un LLM de raisonner dans le vide, vous lui donnez une base structurée : pages, chunks, liens, ancres, embeddings, sitemap, PageRank interne.

Vous pouvez alors piloter votre SEO avec beaucoup plus de précision, notamment sur le maillage interne, la cohérence sémantique et la fraîcheur des données.

Et surtout, vous pouvez le faire de manière incrémentale : seules les pages réellement modifiées sont réembeddées.

C'est ce qui rend le système utilisable dans la durée.
